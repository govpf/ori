# Iconographie

Cette page documente le **choix de la bibliothèque d'icônes** pour Ori, son
intégration dans les trois cibles (React, Angular, HTML statique), et les
règles d'usage.

> **Décision actée (2026-04-25)** : **Lucide** est la bibliothèque officielle
> de Ori, utilisée en mode **strictement monochrome** (color: currentColor,
> jamais de duotone ni multicolore).
>
> Les options évaluées sont conservées plus bas pour traçabilité.

## Règle d'usage monochrome

Toutes les icônes Lucide doivent :

1. **Hériter de la couleur du contexte** via `currentColor` (Lucide le fait
   nativement sur le stroke de chaque icône SVG).
2. **Ne jamais avoir de couleur en dur** : pas de `<svg color="red">`, pas de
   `style="color: #073CA5"` directement sur l'icône. La couleur vient toujours
   du parent (texte, classe sémantique, état).
3. **Ne jamais utiliser duotone ou multicolore**, même si une bibliothèque le
   permettait.

Si une icône doit porter un sens sémantique fort (succès vert, danger rouge,
etc.), c'est le **conteneur** qui porte la couleur, l'icône hérite :

```tsx
{/* ✅ Bon : la couleur vient du conteneur sémantique */}
<span className="text-success">
  <Check />  Tout va bien
</span>

{/* ❌ Mauvais : couleur appliquée directement sur l'icône */}
<Check style={{ color: 'green' }} />
```

Cette règle garantit :
- une **cohérence automatique avec le thème** (light/dark sans intervention)
- une **prédictibilité** : un dev sait que l'icône suit le texte
- un **bundle size optimal** : pas de variantes par couleur

## Pourquoi ce n'est pas trivial

Les icônes sont un **choix structurant** d'un design system :

- elles imposent une **cohérence visuelle** (poids du trait, grille, métaphores)
- elles influencent le **bundle size** (vector font vs SVG individuels)
- elles doivent fonctionner sur les **3 cibles** du DS (React, Angular, HTML pur)
- changer de bibliothèque plus tard est coûteux (chaque app consommatrice doit
  remapper ses icônes)

## Ce qu'utilise pf-ui legacy

Pf-ui mélange **deux bibliothèques** :

| Bibliothèque | Usage dans pf-ui | Pattern |
|---|---|---|
| **PrimeIcons** | ~98% des icônes | `<i class="pi pi-eye"></i>` |
| **Remixicon** | ~2% (cas isolés) | `<i class="ri-checkbox-circle-fill"></i>` |

C'est un **héritage technique**, pas un choix délibéré :

- PrimeIcons était imposé par PrimeNG (icon set bundlé avec la lib)
- Remixicon a été ajouté par-dessus pour combler les manques de PrimeIcons
- Résultat : deux conventions à connaître, deux bundles CSS à charger

Pour Ori, on n'a **plus PrimeNG** → la contrainte PrimeIcons disparaît, on
peut repartir d'une feuille blanche.

## Critères de choix pour Ori

| Critère | Pourquoi c'est important |
|---|---|
| **Multi-framework** (React + Angular + SVG bruts) | Une seule lib pour nos 3 cibles, parité visuelle exacte |
| **Tree-shaking** | Importer seulement les icônes utilisées, pas tout le set |
| **Cohérence visuelle** | Toutes les icônes doivent partager grille, poids, métriques |
| **Couverture** | ≥ 1000 icônes pour couvrir les besoins d'apps admin |
| **License permissive** | MIT/ISC pour pas se retrouver coincé |
| **Communauté active** | Maintenance et ajouts réguliers |

## Options évaluées

### Lucide

| | |
|---|---|
| Volume | ~1600 icônes |
| Style | Trait 2px, grille 24×24, design unifié et professionnel |
| Frameworks | `lucide-react`, `lucide-angular`, `@lucide/web` (web components), SVG bruts via lucide.dev |
| Tree-shaking | Oui (par import) |
| License | ISC |
| Adopté par | shadcn/ui, Vercel Geist, Linear, Cal.com, beaucoup de DS modernes 2024-2026 |

**Pour** : standard de fait moderne, design impeccable, couverture suffisante,
la même icône s'utilise vraiment identiquement entre React, Angular, et un SVG
pasted dans GLPI.

**Contre** : 1600 icônes peuvent ne pas suffire pour des cas très spécifiques
(domaines techniques, illustrations métier).

### Phosphor

| | |
|---|---|
| Volume | ~9000 icônes (1500 × 6 weights : thin, light, regular, bold, fill, duotone) |
| Style | Plusieurs poids, design cohérent dans chaque weight |
| Frameworks | `@phosphor-icons/react`, `@phosphor-icons/web` (web components), Vue, Flutter |
| Tree-shaking | Oui |
| License | MIT |

**Pour** : volume énorme, multiples weights pour des nuances visuelles fines.

**Contre** : pas d'intégration Angular officielle (devra passer par les web
components ou les SVG bruts), design un peu spécifique (parfois "joueur").

### Tabler Icons

| | |
|---|---|
| Volume | ~5800 icônes |
| Style | Trait 2px, grille 24×24, similaire à Lucide |
| Frameworks | `@tabler/icons-react`, `@tabler/icons-angular`, web components, SVG |
| Tree-shaking | Oui |
| License | MIT |

**Pour** : couverture massive (3× Lucide), bonne qualité, multi-framework.

**Contre** : la cohérence est moins parfaite que Lucide sur certains lots
récemment ajoutés (la communauté apporte beaucoup).

### Heroicons

| | |
|---|---|
| Volume | ~300 icônes |
| Style | 2 styles (outline 24×24, solid 20×20) |
| Frameworks | `@heroicons/react`, Vue, SVG |
| Tree-shaking | Oui |
| License | MIT |
| Adopté par | écosystème Tailwind |

**Pour** : design irréprochable, soigné par l'équipe Tailwind.

**Contre** : pas d'intégration Angular officielle, volume limité (300 vs 1600+).

### Remixicon (continuité pf-ui)

| | |
|---|---|
| Volume | ~3000 icônes |
| Style | Variations possibles (line / fill) |
| Frameworks | Font CSS principalement, SVG via téléchargement individuel |
| Tree-shaking | Non (font globale) |
| License | Apache 2.0 |

**Pour** : déjà présent dans pf-ui, transition douce.

**Contre** : pas de composants framework natifs, pas de tree-shaking (toute la
font chargée), design qui commence à dater.

### PrimeIcons (héritage pf-ui)

À écarter sans hésitation : ~250 icônes, design daté, lié à un écosystème
qu'on a quitté, pas de tree-shaking.

## Tableau de synthèse

| Lib | Volume | Multi-framework | Tree-shaking | Design | Recommandation |
|---|---|---|---|---|---|
| **Lucide** | 1600 | React + Angular + SVG | ✓ | Excellent | ⭐ **Top choice** |
| Phosphor | 9000 | React + SVG (Angular via WC) | ✓ | Bon | Si besoin volumineux |
| Tabler | 5800 | React + Angular + SVG | ✓ | Bon | Alternative volumineuse |
| Heroicons | 300 | React + SVG (Angular via WC) | ✓ | Excellent | Si volume suffit |
| Remixicon | 3000 | Font + SVG individuels | ✗ | Daté | Continuité pf-ui mais pas idéal |
| PrimeIcons | 250 | Font | ✗ | Daté | À écarter |

## Recommandation

**Lucide**, pour les raisons suivantes :

1. **Multi-framework natif** : couvre exactement nos trois cibles
   (`lucide-react`, `lucide-angular`, SVG bruts pour HTML pur dans GLPI ou
   sites statiques) avec une parité visuelle parfaite.
2. **Volume suffisant** : 1600 icônes couvrent 99% des besoins d'une app
   admin. Au cas par cas, on peut compléter avec des SVG custom dans le DS.
3. **Tree-shaking par défaut** : `import { Mail, Trash2 } from 'lucide-react'`
   n'embarque que ces deux icônes, pas tout le set.
4. **Design unifié** : grille 24×24 stricte, stroke 2px, alignement pixel
   parfait. C'est la "patte" de tous les DS modernes (shadcn, Linear, Vercel).
5. **License ISC** : équivalent MIT, très permissive.
6. **Communauté très active** : versionnement régulier, ajouts fréquents.

## Plan d'intégration (si Lucide validé)

### Phase 1 - Installation

- Ajouter `lucide-react` aux dépendances de `@govpf/ori-react`
- Ajouter `lucide-angular` aux dépendances de `@govpf/ori-angular`
- Documenter les SVG bruts pour les sites statiques (CDN ou copy-paste)

### Phase 2 - Wrappers DS

Optionnel : créer des composants wrapper Ori qui fixent la taille par défaut
sur les tokens de spacing :

```tsx
// React
<OriIcon name="mail" size="md" />     // size: sm (16px) | md (20px) | lg (24px)
```

```html
<!-- Angular -->
<ori-icon name="mail" size="md"></ori-icon>
```

L'avantage : si un jour on change de bibliothèque, seul le wrapper change.
Inconvénient : couche d'abstraction supplémentaire à maintenir.

À **trancher au moment de l'implémentation** selon l'ergonomie souhaitée.

### Phase 3 - Migration des SVG inline existants

Dans `Alert` et `Dialog`, j'ai actuellement des SVG inline dans le code des
composants. À garder tel quel ou migrer vers Lucide ? Les deux options sont
défendables :

- **Garder inline** : pas de dépendance externe pour le DS de base
- **Migrer vers Lucide** : un seul jeu d'icônes, plus facile à customiser
  uniformément

Recommandation : laisser le DS interne en SVG inline (autonomie), Lucide pour
les apps consommatrices.
