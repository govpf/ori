# FileCard

Le composant **FileCard** affiche un fichier (pièce jointe, document, archive)
sous forme de tuile avec une icône typée, le nom, le type, la taille,
une métadonnée libre, un lien optionnel et jusqu'à quelques actions.

C'est un composant **autonome** : il intègre déjà bordure, padding et ombre
légère. Pas besoin de l'envelopper dans une `<Card>`.

## Pourquoi un composant dédié

Une tuile de fichier est **récurrente** dans les apps administratives PF :
listes de pièces jointes des démarches, espaces documentaires, archives,
signatures, attestations… Sans composant dédié, chaque service risquerait
de produire sa propre version, avec des icônes différentes par type, des
layouts variables et des actions positionnées de manière incohérente.

Le DS porte ces choix une fois pour toutes :

- **mapping `type → icône`** standardisé (un PDF a la même icône partout)
- **layout responsive** validé (le contenu reste lisible même en grille étroite)
- **a11y normée** : `aria-label` automatique sur les actions icône, ordre de tabulation cohérent

## Anatomie

| Région | Rôle |
|---|---|
| `icon` | Pastille avec icône Lucide en fonction du `type` |
| `actions` | Jusqu'à 3 boutons icône en haut à droite |
| `name` | Nom du fichier (ellipsis si trop long) |
| `meta` | Tag du type + texte libre (taille, date) |
| `link` | Lien optionnel en pied de tuile (vers démarche, dossier) |

## Types reconnus

| `type` | Icône | Libellé Tag |
|---|---|---|
| `pdf` | `FileText` | PDF |
| `image` | `Image` | Image |
| `doc` | `FileType` | Document |
| `spreadsheet` | `FileSpreadsheet` | Tableur |
| `archive` | `FileArchive` | Archive |
| `other` | `File` | Fichier |

Le `type` n'est **pas dérivé automatiquement** de l'extension du `name` -
c'est l'app qui doit le passer explicitement, pour éviter les surprises
(un `.txt`, un `.json`, un type MIME exotique…). Si l'app dispose du
type MIME, elle peut le mapper vers l'un des 6 `type` standardisés ; pour
tout le reste, `'other'`.

## Actions

Les actions sont des boutons icône **ghost / size sm** affichés en haut
à droite. Le DS gère :

- l'`aria-label` du bouton, dérivé du `label` de l'action
- le `Tooltip` sur le `label`
- l'icône passée librement (idéalement Lucide pour la cohérence avec le reste du DS)
- la sémantique `data-danger="true"` exposée sur le bouton pour les actions destructives

```tsx
<FileCard
  name="Pièce identité.pdf"
  type="pdf"
  size="1.2 Mo"
  meta="Ajouté le 2026-04-22"
  link={{ label: 'Lié à 2026-0042', href: '#42' }}
  actions={[
    { label: 'Aperçu', icon: <Eye size={16} /> },
    { label: 'Télécharger', icon: <Download size={16} /> },
    { label: 'Supprimer', icon: <Trash2 size={16} />, variant: 'danger' },
  ]}
/>
```

## Bonnes pratiques

- **3 actions max** dans `actions`. Au-delà, regrouper les actions secondaires dans un menu kebab côté app.
- **Garder le nom du fichier exact** dans `name` - y compris l'extension. L'utilisateur a besoin de la voir pour reconnaître son fichier.
- **`size` formaté humainement** (ex: `'1.2 Mo'`, `'850 Ko'`), pas en octets bruts. Le DS n'en fait pas le formatage : c'est du contenu applicatif.
- **`meta` court**. Une date d'ajout, un auteur, un statut. Pas de phrase complète.
- **`link` pour pointer vers le contexte**, pas pour une action sur le fichier. Le téléchargement ou l'aperçu vont dans `actions`.

## Composer une grille

Le composant ne porte pas la grille : c'est l'app qui décide du layout.
Pour rester cohérent avec la demo de référence :

```css
.file-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
  gap: 1rem;
}
```

Le `minmax(280px, 1fr)` est la valeur recommandée : en dessous, le nom de
fichier devient illisible (ellipsis trop agressif et action area trop
étroite).
